# WordPress Transformation Plan — INTERloft
# Deep Technical Analysis & Migration Strategy (Core WordPress Only)

---

## PART 1: SOURCE PROJECT ANALYSIS

### 1.1 Project Overview

- **Type:** Full-stack e-commerce / CMS for luxury Moroccan furniture brand
- **Current Stack:** Next.js 16 (App Router) + React 19 + TypeScript + Supabase + Tailwind CSS v4 + shadcn/ui
- **Architecture:** React Server Components (RSC) + Client Components, file-based routing
- **CMS:** Custom admin dashboard backed by Supabase (no traditional CMS)
- **Auth:** Supabase email/password authentication
- **Content:** Products, Collections, Collaborators, Pages (about, contact, legal)
- **Total routes:** ~25 pages across public, auth, and admin
- **Architecture rating:** 7/10 — solid design system, weak production hardening

## PART 2: CONTENT MAPPING — WORDPRESS CORE ONLY

### 2.1 Content Type Mapping

| Next.js Entity | WordPress Equivalent | Technical Implementation |
|---------------|---------------------|------------------------|
| Products | Custom Post Type | register_post_type('product', ...) |
| Product Categories | Custom Taxonomy | register_taxonomy('product_category', ['product'], ...) |
| Collections | Custom Post Type | register_post_type('collection', ...) |
| Collaborators | Custom Post Type | register_post_type('collaborator', ...) |
| Pages (About, Contact, Legal) | Native WP Pages | wp_insert_post() + page templates |
| Homepage variants | Page Templates | front-page.php, front-page-alt.php |
| User Account | WordPress Users | Native WP users + user profile |
| Auth (Login/Register) | WP Core + Theme Templates | Native WP login (wp-login.php) + custom templates |
| Favorites | Custom Table | wp_favorites table + AJAX |
| Admin CMS | WordPress Admin | Native WP admin + custom meta boxes |

### 2.2 Why Custom Post Types (Not Native Posts)

Native WordPress posts are designed for blog content. CPTs give:
- Clean URL slugs (/product/canape-alba)
- Dedicated admin menus
- Custom columns in admin table
- REST API endpoints auto-generated (/wp-json/wp/v2/product)
- Taxonomies scoped to specific CPTs

## PART 4: CUSTOM META BOXES — NO ACF

### 4.1 Meta Box Architecture

Every Supabase field recreated via add_meta_box() and save_post hooks.

**Meta key naming convention (all prefixed interloft_):**

```
interloft_product_slug          (text)
interloft_product_category      (text — term ID or slug)
interloft_product_collection    (integer — post ID)
interloft_product_description   (textarea)
interloft_product_materials     (array, serialized)
interloft_product_dimensions    (text)
interloft_product_image         (text — attachment URL)
interloft_product_gallery       (array of attachment URLs, serialized)
interloft_collaborator_profession    (text)
interloft_collaborator_city          (text)
interloft_collaborator_hero_image    (text — URL)
interloft_collaborator_expertise     (array, serialized)
interloft_collaborator_story         (textarea)
interloft_collaborator_featured_project (array, serialized JSON)
interloft_collaborator_projects      (array, serialized JSON)
```

### 4.2 Product Meta Box (Product_Meta_Box class)

Three meta boxes: Details (main), Specs (side), Gallery (side).

Key fields: slug, collection relation (dropdown of all collections), main image (URL input + preview), description (textarea), dimensions (text), materials (dynamic add/remove rows), gallery (textarea — one URL per line).

save_meta() handles: text sanitization, integer for collection_id, array serialization for materials and gallery, nonce verification, capability checks.

### 4.3 Collaborator Meta Box (Collaborator_Meta_Box class)

Four meta boxes: Profile (name, profession, city, hero image URL), Story (collaboration story textarea), Projects (featured project + pipe-separated projects list), Expertise (textarea — one material per line).

Featured project stores: name, image URL, description.
Projects stored as pipe-separated: name | image_url | location | description (one per line).

### 4.4 Collection Meta Box (Collection_Meta_Box class)

Fields: description (textarea), image (URL input with preview), order (integer for display sorting).

### 4.5 Page Meta Box (Page_Meta_Box class)

Attached only to specific pages (introduction, contact) by checking post_name. Fields: subtitle, extra_info.

## PART 5: FAVORITES SYSTEM — TWO OPTIONS

### 5.1 Option 1: User Meta (Simple, Limited)

```php
$favorites = get_user_meta($user_id, 'interloft_favorites', true) ?: [];
// Add: $favorites[] = $product_id; update_user_meta(...);
// Remove: array_filter(...)
```

Problem: serialized string, lock contention at scale, ~100 item practical limit.

### 5.2 Option 2: Custom Table (Recommended)

```sql
CREATE TABLE wp_favorites (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,
    product_id BIGINT UNSIGNED NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY unique_favorite (user_id, product_id),
    KEY idx_user (user_id),
    KEY idx_product (product_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

Class Interloft_Favorites with methods:
- create_table() — runs on theme activation via register_activation_hook
- toggle_favorite(user_id, product_id) — INSERT or DELETE
- get_favorites(user_id) — returns array of product_ids
- is_favorite(user_id, product_id) — boolean check
- ajax_toggle_favorite() — wp_ajax handler
- ajax_get_favorites() — wp_ajax handler

Frontend: Vanilla JS fetch() to admin-ajax.php with nonce verification.

## PART 7: WORDPRESS REST API — NO PLUGINS

### 7.1 Extended Product Endpoints

register_post_meta with show_in_rest => true for every meta key.

register_rest_field adds computed fields:
- product_category_name (get_the_terms first term name)
- product_category_slug (first term slug)

### 7.2 Custom Search Route

register_rest_route('interloft/v1', '/search', ...):
- GET parameter: q (search query)
- Searches products via WP_Query with s parameter (full-text search)
- Searches product_category terms via get_terms with name__like
- Returns: { products: [...], categories: [...] }
- permission_callback: __return_true (public)

### 7.3 Frontend Search JS

Vanilla JS with debounce (250ms). Fetches from /wp-json/interloft/v1/search?q={query}. Renders results in search overlay with product cards and category buttons.

## PART 8: AUTHENTICATION — WORDPRESS CORE

### 8.1 Login (page-connexion.php)

wp_signon() for credential verification. On success: wp_redirect to /compte/. On failure: display error. Logged-in users redirected to /compte/.

### 8.2 Registration (page-inscription.php)

wp_create_user(email, password, email) for new user creation. wp_update_user to set first_name, last_name, role='subscriber'. wp_signon to log in immediately after registration.

### 8.3 Account Page (page-compte.php)

wp_get_current_user() for data. wp_update_user on form submission with nonce verification. Favorites loaded via AJAX from wp_favorites table. Logout via wp_logout_url().

## PART 9: FRONTEND COMPONENTS — PHP TEMPLATES

### 9.1 Navbar (header.php)

Three sections: mobile menu button (left), desktop nav with dropdowns (left), centered logo, search + account icons (right). Dropdowns fetch categories via get_terms() inline or cached. Mobile overlay toggled via main.js. Search overlay with input and results container.

### 9.2 Product Card (template-parts/product-card.php)

Shows: product image (from interloft_product_image meta), category label (from first term), title linking to product URL (built from category slug + product slug meta), favorite button (if logged in, checked via is_favorite()).

### 9.3 Contact Form (page-contact.php)

AJAX submission to wp_ajax_submit_contact. Fields: name, email, subject, message. Sanitization + validation in handler. wp_mail() sends to contact@interloft.ma. Message saved to wp_contact_messages table. Success/error displayed via JS.

## PART 11: LOCAL WORDPRESS SETUP (NO DOCKER)

### 11.1 Environment: Laragon (Recommended)

**Step 1:** Download Laragon Full from https://laragon.org/download/

**Step 2:** Install → Run Laragon → Click Terminal

**Step 3:** Create project folder and download WordPress:
```bash
cd C:\laragon\www
mkdir interloft
cd interloft
curl -o wordpress.zip https://wordpress.org/latest.tar.gz
unzip wordpress.zip
mv wordpress/* .
rmdir wordpress wordpress.zip
```

**Step 4:** Create database
```
http://localhost/phpmyadmin
→ New → Name: interloft_db → Collation: utf8mb4_unicode_ci → Create
(credentials: user=root, password=empty)
```

**Step 5:** Configure wp-config.php:
```php
define('DB_NAME', 'interloft_db');
define('DB_USER', 'root');
define('DB_PASSWORD', '');
define('DB_HOST', 'localhost');
define('DB_CHARSET', 'utf8mb4');
define('DB_COLLATE', '');
// Add security keys from https://api.wordpress.org/secret-key/1.1/salt/
define('WP_DEBUG', true);
define('WP_DEBUG_LOG', true);
define('WP_DEBUG_DISPLAY', false);
define('WP_HOME', 'http://interloft.test');
define('WP_SITEURL', 'http://interloft.test');
$table_prefix = 'wp_';
```

**Step 6:** Complete installation at http://interloft.test (Laragon auto-creates virtual host)

**Step 7:** Create theme:
```bash
cd C:\laragon\www\interloft\wp-content\themes
mkdir interloft-theme
```

**Step 8:** Add style.css with theme header, activate in WordPress Admin

**Step 9:** Set permalinks: Settings → Permalinks → Post name

### 11.2 Alternative: XAMPP

Same steps as above, but extract WordPress to C:\xampp\htdocs\interloft\ and access at http://localhost/interloft/. Add virtual host manually in httpd-vhosts.conf.

## PART 14: PERFORMANCE & SEO

### 14.1 Caching (Hostinger production)

OPcache in php.ini: opcache.enable=1
Browser caching via .htaccess: ExpiresByType for images (1 year), CSS/JS (1 month)
Gzip via mod_deflate

### 14.2 Image Optimization

WordPress 5.5+ auto-adds loading="lazy". Remove unused sizes via intermediate_image_sizes filter. Alt text from product name via the_title_attribute().

### 14.3 Manual SEO

In header.php: wp_title(), canonical links, Open Graph meta tags, og:image from product meta.
JSON-LD in single-product.php: Product schema with name, description, image, url, brand, offers.

### Key Risks

1. Auth UX gap vs Supabase's polished auth flow
2. Performance regression (dynamic PHP vs static Next.js)
3. Image migration from Supabase Storage to WordPress Media Library
4. Favorites AJAX requires careful nonce/error handling
5. Meta box maintenance burden for future schema changes

### Recommendation

Two options:

**Option A — Full WordPress (detailed above):**
Complete PHP rewrite. Best if client explicitly needs WordPress admin for non-technical content managers.

**Option B — Headless WordPress (Recommended):**
Keep Next.js as frontend, WordPress as headless CMS via REST API. Preserves entire design system, all components, TypeScript, React hooks. Only WordPress CPTs + meta boxes needed (no theme files). Industry-standard approach for projects like this.
