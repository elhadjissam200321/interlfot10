# Technical Audit Report — INTERloft Next.js Project
## Headless WordPress Integration Readiness Assessment

---

## EXECUTIVE SUMMARY

The codebase has a solid architectural foundation (7/10) with modern Next.js 16, React Server Components, and a clean design system. However, before integrating with WordPress as a headless CMS, **7 critical security issues must be fixed**. The most urgent is that `/admin/*` routes are **NOT protected by middleware** — anyone can access the admin panel without login.

The project also has a `next.config.mjs` with `ignoreBuildErrors: true` that silently hides type errors in production builds. Removing this and fixing the 4 TypeScript errors is a prerequisite for a clean headless integration.

---

#### C1: Admin Routes Have No Middleware Protection

**File:** `middleware.ts:53-63`

**Problem:** The middleware matcher explicitly excludes `/api/*` routes but **does NOT include `/admin/*`**. The middleware only protects `/compte` and `/protected` paths:

```typescript
if (
  (request.nextUrl.pathname.startsWith('/compte') ||
   request.nextUrl.pathname.startsWith('/protected')) &&
  !user
) {
  return NextResponse.redirect(url)
}
```

This means anyone can navigate directly to `/admin`, `/admin/products`, `/admin/products/new` without authentication. The admin layout has a server-side `getUser()` check, but this is a **defense-in-depth** issue — middleware should be the first line of defense.

**Impact:** Any visitor can access the admin dashboard, view all products/categories, and trigger the migration tool.

**Fix:**
```typescript
// middleware.ts — add /admin to protected paths
if (
  (request.nextUrl.pathname.startsWith('/compte') ||
   request.nextUrl.pathname.startsWith('/admin') ||
   request.nextUrl.pathname.startsWith('/protected')) &&
  !user
) {
  const url = request.nextUrl.clone()
  url.pathname = '/auth/connexion'
  return NextResponse.redirect(url)
}
```

#### C3: XSS via ColorSwitcher localStorage Manipulation

**File:** `components/color-switcher.tsx:116-127`

**Problem:** The color switcher loads a hex value from `localStorage`, validates it with `isValidHex()`, and sets it as a CSS custom property on `document.documentElement`. However, the `localStorage` value is stored as `JSON.stringify({ 'bg-page': hex })`:

```typescript
const saved = localStorage.getItem(STORAGE_KEY)
if (saved) {
  const parsed = JSON.parse(saved)  // parsed value is validated
  const hex = typeof parsed === 'string' ? parsed : parsed['bg-page'] || SLOT_DEFAULT
  if (isValidHex(hex)) {  // validation passes
    document.documentElement.style.setProperty(SLOT_VAR, hex)
  }
}
```

The code validates the hex, but an attacker who can write to localStorage (via another XSS, browser extension, or shared computer) could inject:
- CSS injection via long hex values: `calc(1)` can lead to scrollbar attacks
- Property injection: `; --fake-var: url("javascript:alert(1)")` (modern browsers prevent `javascript:` in CSS, but other injections are possible)

**Impact:** CSS injection for UX defacement. Limited scope but still a vulnerability.

**Fix:** Add a strict allowlist of hex values, ignore any custom input that isn't in the predefined palette:

```typescript
const PALETTE_HEXES = new Set(BEIGE_PALETTE.map(p => p.hex))

function handleChange(hex: string) {
  if (!isValidHex(hex)) return
  // Only allow palette colors for stored values
  if (!PALETTE_HEXES.has(hex)) return
  setColor(hex)
  document.documentElement.style.setProperty(SLOT_VAR, hex)
  localStorage.setItem(STORAGE_KEY, JSON.stringify({ 'bg-page': hex }))
}
```

Also, the navbar reads `--color-navbar` and `--color-mobile-menu` from a CSS variable:

```typescript
// navbar.tsx:50
style={scrolled ? { backgroundColor: 'var(--color-navbar)' } : undefined}
// navbar.tsx:142
style={{ backgroundColor: 'var(--color-mobile-menu)' }}
```

These CSS vars are set by the ColorSwitcher but not validated. Ensure only palette hexes are used.

#### C5: Supabase RLS Policies Unknown

**Files:** All queries in `lib/supabase/queries.ts`

**Problem:** The Supabase client uses the anon key, meaning queries go through RLS (Row Level Security) policies. However, the actual RLS policies are defined in `lib/supabase/schema.sql` which was not provided. If RLS is not configured, the public could read/write all data.

**Impact:** Unknown — depends on RLS configuration. If not set, full database access to everyone.

**Action:** Verify in Supabase dashboard:
1. RLS is enabled on all 5 tables (products, categories, collections, collaborators, pages_content)
2. Public read: `SELECT` for everyone (`(auth.role() = 'anon')`)
3. Authenticated write: `INSERT`, `UPDATE`, `DELETE` only for authenticated users
4. Admin write: `UPDATE`, `DELETE` only for admin users

#### H3: Mobile Menu Color from localStorage Without Sanitization

**File:** `components/navbar.tsx:50, 142`

```typescript
style={scrolled ? { backgroundColor: 'var(--color-navbar)' } : undefined}
// navbar.tsx:142
style={{ backgroundColor: 'var(--color-mobile-menu)' }}
```

**Problem:** The navbar uses CSS custom properties `--color-navbar` and `--color-mobile-menu` set by the ColorSwitcher, but these are not validated. If `--color-mobile-menu` is set to a malicious value, it could break the mobile menu overlay.

**Fix:** Ensure ColorSwitcher only sets validated palette colors, OR sanitize the CSS variable reads:

```typescript
function sanitizeCSSColor(color: string): string {
  // Allow only safe CSS color values
  if (/^#[0-9A-Fa-f]{6}$/.test(color)) return color
  return 'var(--background)'  // fallback to safe default
}
```

#### H4: Auth Pages Are Unnecessarily Client Components

**Files:** 
- `app/auth/connexion/page.tsx`
- `app/auth/inscription/page.tsx`
- `app/auth/mot-de-passe-oublie/page.tsx`

**Problem:** These three pages are `'use client'` but only use `useState` for form inputs. The Supabase `signInWithPassword` and `signUp` calls could be done in a Server Action instead, eliminating the client bundle entirely.

**Impact:** Larger client bundle, slower initial load, unnecessary hydration.

**Fix — Convert to Server Actions:**
```typescript
// app/actions/auth.ts
'use server'
export async function login(formData: FormData) {
  const supabase = await createClient()
  if (!supabase) return { error: 'Configuration error' }
  
  const { error } = await supabase.auth.signInWithPassword({
    email: formData.get('email') as string,
    password: formData.get('password') as string,
  })
  
  if (error) return { error: error.message }
  redirect('/compte')
}
```

Then the page becomes a Server Component with a `<form action={login}>`.

#### H5: Search Modal Directly Queries Supabase from Client

**File:** `components/search-modal.tsx:70-86`


#### H6: Favorites Stored in User Metadata — Scalability Limit

**File:** `hooks/use-favorites.ts:19-20`

```php
const userFavorites = user.user_metadata?.favorites || []
```

**Problem:** 
1. User metadata is serialized — every toggle reads the entire favorites array, modifies it, and writes it back
2. With 100+ favorites and 1000+ concurrent users, this causes database lock contention
3. No dedicated favorites table in Supabase

**Impact:** Will break at scale. Favorites will be lost or corrupted under load.

**Fix — Create dedicated favorites table:**
```sql
CREATE TABLE favorites (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID REFERENCES auth.users NOT NULL,
  product_id TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, product_id)
);

-- RLS: Users can only read/insert/delete their own favorites
ALTER TABLE favorites ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users manage own favorites" ON favorites
  FOR ALL USING (auth.uid() = user_id);
```

Then update `use-favorites.ts` to query this table instead of user metadata.

#### M1: No Pagination on Product Lists

**Files:** 
- `app/products/page.tsx`
- `app/admin/products/page.tsx`
- `hooks/use-products.ts:30`

```typescript
const { data } = await supabase.from('products').select('*')
// No .limit() or pagination
```

**Problem:** All products are fetched at once. With 1000+ products, this causes slow loads and memory issues.

**Fix:** Add pagination with `range(from, to)`:
```typescript
const { data } = await supabase
  .from('products')
  .select('*', { count: 'exact' })
  .range(page * limit, (page + 1) * limit - 1)
```

#### M5: Inconsistent Error Handling in Queries

**Files:** `lib/supabase/queries.ts`

Some functions have try/catch:
```typescript
// getProducts — has try/catch
// getProductsByCategory — NO try/catch
// getCategoryBySlug — NO try/catch
```

**Fix:** Standardize all query functions with consistent error handling:
```typescript
export async function getProductsByCategory(category: string) {
  try {
    const supabase = await createClient()
    if (!supabase) return []
    const { data, error } = await supabase
      .from('products').select('*').eq('category', category)
    if (error) {
      console.error('getProductsByCategory error:', error)
      return []
    }
    return data || []
  } catch (e) {
    console.error('getProductsByCategory exception:', e)
    return []
  }
}
```
